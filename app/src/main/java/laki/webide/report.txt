Laki Web IDE - File Handling System Consolidation Report

1. Current Architecture Complexity
The file system is fragmented across multiple classes, causing redundant path calculations and inconsistent save/load locations.

A. LakiFiles.java (Lines 9-512)
   - Role: Define folder names and relative path templates.
   - Methods: getPageExtendBlocksPath (L387), getPageHtmlTagsPath (L395).
   - Problem: These methods require a 'projectRoot' to be passed in, but the class doesn't handle the resolution of that root using sc_id.

B. FilePathUtil.java (Lines 1-85)
   - Role: Middleman resolver that links sc_id to LakiFiles.
   - Methods: getProjectRoot (L12-24), getPathAssets (L49).
   - Problem: Redundant layer. It performs a metadata lookup (lC.b(sc_id)) just to call LakiFiles. This lookup should be inside LakiFiles.

C. ProjectWorkspace.java (Lines 55-94)
   - Role: Logic Manager / Context.
   - Problem: The constructor manually calculates projectMyscPath, assetsPath, etc. It also contains heavy 'Migration' logic (L81-92) that moves files from legacy folders. This makes the workspace initialization slow and prone to errors.

D. CssLogicPersistenceManager.java (Lines 54-137)
   - Role: JSON persistence for blocks.
   - Problem: Manually performs metadata lookups and root resolution again. Contains 'Extension Fallback' logic (L111-118) to guess if a file is .java or .css, which is no longer needed with the new unified naming.

2. Consolidation Plan
To simplify the code and ensure reliable save/load, we should:

- Delete FilePathUtil.java: Move all its logic into LakiFiles.java.
- Simplify LakiFiles.java:
  - Add a private helper: resolveMetadata(sc_id) to get projectName.
  - Change all methods to accept sc_id directly: e.g., getHtmlPath(sc_id) instead of getHtmlPath(projectRoot).
- Clean ProjectWorkspace: Should only hold sc_id. All paths should be retrieved via LakiFiles.get...(sc_id).
- Clean Persistence Managers: Remove all extension guessing and root resolution. Use direct calls like LakiFiles.getPageExtendBlocksPath(sc_id, pageName).

3. Impact
- Reducing 4 files of path logic down to 1 (LakiFiles).
- Centralized Migration: Move migration logic out of constructors into a one-time LakiFiles.initProject(sc_id) call.
- Guaranteed Consistency: The editor and compiler will always look at the exact same folder string.

Part 2: Java NullPointerException Analysis
-----------------------------------------
Error: java.lang.NullPointerException: Attempt to invoke virtual method 'boolean java.lang.String.equals(java.lang.Object)' on a null object reference
Location: a.a.a.eC.a(...) -> ViewEditorFragment.a (L189) -> i (L335)

1. Root Cause:
The crash occurs during project loading when the Sketchware engine tries to "Clone" the list of ViewBeans for the current page. The method eC.a() iterates through the list of widgets and performs a .equals() on their IDs or parent references.

2. Trigger:
This was likely triggered by the new loading logic in ViewEditorFragment.i(). When the IDE reads the visual history from settings/[page]_htmltags.json, if the JSON contains a null element or a ViewBean with a missing "id" or "parent" field, the internal Sketchware cloner (eC) fails.

3. Specific Method Trace:
- ViewEditorFragment.i() (Line 335): Calls b(jC.a(sc_id).d(...)) to load page widgets.
- ViewEditorFragment.a() (Line 189): Calls viewEditor.a(eC.a(viewBeans)).
- eC.a(): Fails at line 25 because it encountered a null String when comparing widget IDs.

4. Required Fix:
- Add a "Sanity Check" in ViewEditorFragment.i() after loading from JSON.
- Filter out any null entries or beans with null IDs from the viewBeans list before putting them into the jC.a(sc_id).c map.
- Ensure the "root" widget (ID: root) is always present and correctly initialized if missing from the JSON.
